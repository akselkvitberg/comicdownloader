## Key Decisions

### Web Framework
F# used ASP.NET Core with attribute routing. Go uses the stdlib `net/http` mux introduced in Go 1.22, which supports method+path patterns (`GET /health`, `POST /jobs/download`). No framework dependency added.

### Image Deduplication
F# used ImageSharp's RGBA32 pixel buffer via `CopyPixelDataTo`. Go uses `image.Decode` from the standard library, iterating pixels with `img.At(x, y).RGBA()`. Both produce a base64-encoded MD5 of pixel data, but the pixel encoding differs slightly for JPEGs.

**Impact:** Existing GCS hash markers written by the F# service will not match Go's hashes. Each comic re-downloads exactly once after deploy, then the new Go hash is stored. This is harmless.

### OneDrive
F# used the Microsoft.Graph SDK. Go uses direct HTTP calls to the Microsoft Graph REST API (no SDK). The chunked upload uses 320 KB chunks for files over 4 MB, matching the Graph API's required alignment.

### Telegram Chat ID
F# stored `TelegramSettings.User` as a string in Secret Manager. The Go Telegram bot API requires `int64`. `NewTelegramClient` parses the string with `strconv.ParseInt` at client construction time â€” no secret format change needed.

### TU Comics URL
The F# source produced `?namelunch` (no `=` between the key `name` and the comic value `lunch`). This appears to be a bug in the original F# string interpolation (`$"...?name{comic}&date=..."`), but since the API accepts it, the behavior was replicated faithfully in Go.

### GCP Clients
Go uses Application Default Credentials via the official `cloud.google.com/go` SDKs, matching the F# ADC setup. The `AppContext` struct holds both the Storage and Secret Manager clients together with the project/bucket configuration.

### Logging
F# used Microsoft.Extensions.Logging. Go uses `log/slog` with a JSON handler, producing structured JSON logs compatible with Cloud Logging.

## Issues Encountered

**Secret Manager import path** â€” The plan referenced `cloud.google.com/go/secretmanager/apiv1/apiv1pb` but the correct package at v1.14.6 is `secretmanagerpb`. Fixed by the implementer.

**Invalid test PNG** â€” The 1Ă—1 PNG bytes in the plan's test fixture had an invalid IDAT checksum. The implementer replaced them with bytes generated by Go's own `image/png` encoder.

**Unhandled `http.NewRequest` errors in onedrive.go** â€” All three `http.NewRequest` call sites discarded the error with `_`. Fixed during code quality review.

**Chunk response body not drained** â€” In the chunked upload loop, `chunkResp.Body.Close()` was called without draining first, preventing TCP connection reuse. Fixed by adding `io.Copy(io.Discard, chunkResp.Body)` before `Close`.

**`go mod tidy` version bumps** â€” MVS bumped `cloud.google.com/go/storage` from v1.47.0 to v1.50.0 and `go 1.22` to `go 1.23.0`. Accepted as normal Go module behavior.

## Container

The Dockerfile uses a two-stage build:

1. `golang:1.22-alpine` â€” compiles a statically linked binary (`CGO_ENABLED=0`)
2. `gcr.io/distroless/static:nonroot` â€” minimal runtime image (~3 MB base)

The resulting image is significantly smaller than the .NET-based image (~20 MB vs ~200 MB+).
